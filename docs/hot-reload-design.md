# 热更新系统设计文档

## 📋 需求分析

基于当前项目结构，需要实现：
1. **文件变化检测**：`tstl --watch` 检测 TypeScript 文件变化
2. **外部通知机制**：`dev.ts` 通知游戏内部哪个模块需要重新加载
3. **游戏内热重载**：游戏内部接收通知并重新加载指定模块

## 🏗️ 系统架构

```
[TypeScript 文件变化] 
         ↓
[TSTL 自动编译到 dist/]
         ↓
[dev.ts 检测编译完成]
         ↓
[生成 hot-reload.json 通知文件]
         ↓
[游戏内定时器读取通知文件]
         ↓
[重新加载指定模块]
         ↓
[更新游戏状态]
```

## 📁 文件结构设计

```
dist/
├── src/                    # 编译后的 Lua 文件
│   ├── main.lua
│   ├── system/
│   └── utils/
├── hot-reload.json         # 热更新通知文件
└── map.w3x

src/
├── system/
│   └── HotReload.ts        # 游戏内热更新管理器
└── main.ts                 # 主程序（注册热更新）
```

## 🔧 通知机制设计

### 1. 通知文件格式 (`dist/hot-reload.json`)

```json
{
  "timestamp": 1697123456789,
  "action": "reload",
  "modules": [
    "src.system.actor",
    "src.utils.CameraControl"
  ],
  "processed": false
}
```

### 2. 外部通知器 (`dev.ts` 扩展)

- 监听 TSTL 编译完成事件
- 分析变化的文件，转换为模块路径
- 生成通知文件到 `dist/hot-reload.json`

### 3. 游戏内热更新管理器 (`src/system/HotReload.ts`)

- 定时器轮询读取通知文件
- 解析需要重新加载的模块
- 执行模块重新加载逻辑
- 标记通知为已处理

## 📋 技术细节

### 模块路径转换规则
```
src/system/actor.ts     → src.system.actor
src/utils/helper.ts     → src.utils.helper
src/main.ts            → src.main
```

### 模块重新加载策略
1. **清除模块缓存**：`package.loaded[module_name] = nil`
2. **重新 require**：`local new_module = require(module_name)`
3. **状态迁移**：保持必要的游戏状态
4. **事件通知**：通知相关系统模块已更新

### 性能考虑
- 轮询间隔：0.5秒（避免过于频繁）
- 文件大小限制：< 1KB（避免大文件解析）
- 模块缓存：智能清除相关依赖

## 🔒 安全和稳定性

### 错误隔离
- 热更新失败不影响游戏运行
- 模块加载错误时回退到旧版本
- 提供手动禁用热更新的开关

### 状态一致性
- 保持关键游戏状态（单位、UI等）
- 重新初始化时的状态恢复
- 避免内存泄漏
